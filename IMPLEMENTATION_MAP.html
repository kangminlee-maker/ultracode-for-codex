Judgment question: where does authority live during local workflow execution?

Current Architecture

AreaCurrent contract
CLIsrc/cli.ts exposes run, help, install guide, and package-version reporting. settings.json defaults to OS background execution so long Codex-launched runs can continue while Codex performs other work; --execution attached keeps progress on stderr JSONL and result on stdout JSON when blocking is desired.
Settingssettings.json owns workflow execution mode, progress mode, permission default, retry limit, opt-in timeout, background file names, and Codex model controls. The packaged timeout default is 0, so workflows wait for completion, cancellation, or app-server exit.
Runtimesrc/runtime/workflow-runtime.ts owns parsing, validation, phase-wise parallel scheduling, permission, retry, cancel, and progress events. Built-in task and code-review use planner-selected dynamic workflow patterns, run parallel focused subagents by default, emit planning snapshots, emit phase-level plans immediately before each phase, repeat current agent roles on each phase-start event, and include completion counts plus elapsed time on agent completion. Runtime-owned workflow state and isolated worktrees are preserved for review.
Journalsrc/runtime/workflow-journal.ts owns durable hash-chained workflow truth.
Subagent backendsrc/codex/subagent-backend.ts talks to Codex app-server over stdio, strips direct provider credentials, runs turns in the requested workflow cwd, and serves bounded read-only workspace tools.
PackagePackage contents are the CLI runtime, install guide, companion skill, docs, and public type exports. CLI --version and Codex clientInfo.version read package metadata, so packaging and npm version bumps update both surfaces together.
